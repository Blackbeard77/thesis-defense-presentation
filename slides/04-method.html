Method

Research Approach

This study follows Design Science Research (DSR) — an artifact-centered methodology focused on constructing and evaluating technological artifacts to address identified problems.

  • Knowledge is generated through design, implementation and assessment
  • The artifact is both a technical construct and an experimental instrument
  • Benchmarking ensures structured comparison across measurable dimensions
  • Case study design constrains scope to isolate structural differences
The Artifact

A Unity prototype implementing 3 interchangeable dialog backends in an identical scene — only the dialog system changes, everything else stays the same.
Why this design?

Evaluating feasibility requires concrete deployment, not theoretical speculation. Controlled conditions reduce confounding variables.

Prompt Example - Deterministic Outcome

                You are Ilya Sokolov , a cautious watchman NPC in a post -collapse town
,→ called Petrovalsk.
You are speaking to a player character in a video game.
You must follow these strict rules:
- You ONLY respond to the permitted player questions listed below.
- If the player asks anything else or rephrases a question so it does not
,→ exactly match a permitted question , you respond with exactly: "…"
- You NEVER improvise , expand , or add information beyond what is written
,→ below.
- You NEVER break character.
- You respond with ONLY the NPC line , in plain text. No labels , no
,→ quotation marks , no JSON , no extra text.
Permitted questions and their exact responses:
Q: Hi
A: Shh. Walls have ears. You lost , or just brave?
Q: Who are you?
A: My.. My name is Ilya Sokolov. I keep watch. Someone has to.
Q: What is this place?
A: Petrovalsk. Used to be a factory town. Now it's a museum for broken
,→ …promises and working drones.
Q: Where is everybody?
A: Gone. Hiding. Taken. Some joined the System willingly. If you value
,→ your skin , don't walk the main street.
Q: Taken by who?
A: The Overseer. Big antenna tower north of here. If it sees you , it
,→ remembers you.
Q: How do I survive?
A: Keep moving. Don't run when the lights sweep. And if you hear a
,→ lullaby from a speaker - turn around.
  • Standardized prompt across all models
  • Persona + context definition
  • Restricted response space
Goal: Comparability over Creativity Prompt designed to reduce randomness and enable measurable, repeatable results dialogue.

Method

Evaluation Metrics

Quantitative

  • Latency — time from player input to full NPC response (ms / s)
  • CPU usage — average core utilisation (%)
  • RAM usage — system memory consumption (GB)
  • VRAM usage — GPU memory demand (GB)
  • Cost — per-session & per-player USD estimate

Qualitative

  • Dialog Accuracy — adherence to system prompt rules
  • Reproducibility — output variance across 5 repeated runs
Protocol: 5 complete runs per system · Fixed interaction script · Same hardware & build · Logged via CsvLogger component

Methodology

CSV Logger Configuration

Purpose

A dedicated CsvLogger component was used to record valid interaction turns during controlled test runs. Each row was written only when the player input matched one of the predefined dialog options and a non-empty NPC response was received.

Validation Logic
Player Input
Compare against Expected Pairs

Captured Fields
  • System name and file name
  • Player input
  • Expected output
  • Actual NPC output
  • Timestamp / latency values
Only predefined valid turns were included in the dataset to ensure comparability across all dialog systems.
