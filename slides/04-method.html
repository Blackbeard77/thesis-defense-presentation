Method

Research Approach

This study follows Design Science Research (DSR) — an artifact-centered methodology focused on constructing and evaluating technological artifacts to address identified problems.

  • Knowledge is generated through design, implementation and assessment
  • The artifact is both a technical construct and an experimental instrument
  • Benchmarking ensures structured comparison across measurable dimensions
  • Case study design constrains scope to isolate structural differences
The Artifact

A Unity prototype implementing 3 interchangeable dialog backends in an identical scene — only the dialog system changes, everything else stays the same.
Why this design?

Evaluating feasibility requires concrete deployment, not theoretical speculation. Controlled conditions reduce confounding variables.

Prototype & Scenario Design

The scenario takes place in Petrovalsk — a fictional post-apocalyptic Soviet town. The player interacts with NPC Ilya Sokolov.

6 Dialog Options

  • Hi
  • Who are you?
  • What is this place?
  • Where is everybody? unlocks 2 branches

Why controlled?

All three implementations use the identical scene: same assets, lighting, character models, UI layout, and dialog objectives. Only the backend logic differs.

Evaluation Metrics

Quantitative

  • Latency — time from player input to full NPC response (ms / s)
  • CPU usage — average core utilisation (%)
  • RAM usage — system memory consumption (GB)
  • VRAM usage — GPU memory demand (GB)
  • Cost — per-session & per-player USD estimate

Qualitative

  • Dialog Accuracy — adherence to system prompt rules
  • Persona consistency — does Ilya stay in character?
  • Reproducibility — output variance across 5 repeated runs
Protocol: 5 complete runs per system · Fixed interaction script · Same hardware & build · Logged via CsvLogger component
