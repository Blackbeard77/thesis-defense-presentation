Runtime Behaviour & Performance

Each system showed a distinct performance profile under identical conditions:

Dialog Tree
Flat CPU, near-zero latency. All responses retrieved from pre-authored structures — no computation at runtime.
Cloud LLM (OpenAI)
Low local CPU — inference runs externally. Latency comes from network + generation time. Visible response windows in the graphs.
Local LLM
High CPU spikes during generation. Latency comparable to cloud for small models. Larger models cause sustained CPU saturation — incompatible with real-time dialog.
The sweet spot: Gemma 3 (1B) achieved latency comparable to OpenAI while running entirely offline — the best local trade-off.

Tradeoffs

Dialog Tree

Predictable Zero cost Fast
Rigid High authoring No scalability

Best for small, well-defined interaction scenarios where every path can be fully enumerated.

Local LLM

No API cost Private Flexible
High hardware req. Prompt engineering

Best when moderate flexibility is needed without external dependencies or recurring costs.

Cloud LLM

Most capable Low local HW Best accuracy
Recurring cost Network dep. Vendor lock-in

Best when maximum capability justifies the additional cost and external service reliance.

Important: No single "best" solution exists. The right choice depends on game genre, team size, budget, and desired player experience.

Limitations

  • Single NPC, small scenario — findings may not directly generalise to open-world settings or large NPC casts
  • No concurrent sessions tested — the "breaking point" for local LLMs under parallel dialog load remains unknown
  • No user study — player experience and perceived dialog quality were not formally evaluated
  • Pre-trained models only — fine-tuned or game-specific models were outside the scope of this work
  • No hybrid architectures tested — RAG, multi-agent setups, and tree + LLM combinations were not explored
Scope reminder

This study was intentionally focused on a controlled prototype condition — one NPC, six dialog options, five runs.

The goal was a developer-oriented comparison framework, not a universal prescription for all game types.

These limitations are natural next steps for future research — outlined in the Conclusion.
